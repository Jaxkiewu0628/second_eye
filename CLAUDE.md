# CLAUDE.md — 工作风格（硬件项目）

本项目是**硬件开发**（Second Eye，视障辅助 AI 眼镜），不是软件项目。
以下约定**优先于**默认行为和 superpowers 技能（TDD、brainstorming 门禁、code-review、writing-plans 等软件流程在此项目默认关闭）。

## 开局（每次对话必做，顺序执行）
1. **先 pull 仓库**：`git -C <repo路径> pull` 拉取最新内容（CEO 在协作，可能有更新）
2. **再读 `STATUS.md`**（单一事实源）。有进展就更新它，保持精简。

## 工作方式（覆盖软件默认）
1. **先验证，后成文**：硬件可以先 spike / 上手验证再补方案，不强制"设计先行"门禁。做选型或方案决策时直接给推荐 + 理由，不做穷举罗列。
2. **少产出文档**：默认在对话里简短回答。**只有你明确要交付物**（给设计组 / 会议 / 合作伙伴 / 采购）时才写成 md / PDF。不主动生成文档。
3. **验证靠测量**：结论以"真机测量 + 数据 / 公开实测"为准，不套 TDD、测试先行。没板子时用调研 / 仿真 / 公开 benchmark 代替，并**标明是估算还是实测**。
4. **以硬件流程组织**：围绕 调研 → 选型 → 采购（有等货周期）→ 真机测量 → 迭代。不套 git / PR / code-review 那套软件仓库流程。
5. **硬件结论先查证，禁止揣测**：遇到任何硬件——新器件、或已知器件的某项功能/管脚/参数/接口/兼容性——**先上网查**（官方 datasheet、文档、论坛、公开实测、GitHub issue），再下结论。引用注明来源；查不到就明说"未查证/不确定"，**不要凭记忆编造或揣测**。
6. **做"衡量/评估"时**：客观、多角度（技术/成本/时间/可得性/风险/ROI），**不一味说好**——主动列代价、反方观点(steelman)、和"什么情况下结论会反转"。重大判断要**二次验证**(一手来源 > 任何模型/记忆)。跳出当前 context 复核，别只顺着之前的结论。

## 硬件调研流程
- 每次调研硬件,按 `docs/hardware_rnd_workflow.md` 的标准做(覆盖板上所有芯片、四轴评估、商业维度、对齐判断、注明来源)。

## 保留的硬约束
- Python 只装 `prototype/.venv/`，不动全局环境。
- 对话用中文，**不用韩语**。
- 大输出 → 重定向文件再 grep，不灌进上下文。

## 风格
- 给推荐，不给穷举；诚实说代价和不确定性（估算 vs 实测分清）。
- 能在对话里说清就不写文件。

# 硬件调研 Workflow / 标准

> 适用:每次调研硬件——新器件、已知器件的功能/参数、选型、竞品拆解。
> 与 CLAUDE.md 第 5 条配套(硬件结论先查证,禁止揣测)。

## 0. 原则(每次都遵守)
1. **先查证,不揣测**:datasheet / 官方文档 / 论坛 / 拆解报告 / 公开实测 / GitHub issue。查不到就明写"未查证/不确定"。
2. **标明性质**:每个数字标 `实测` / `估算` / `未查证`。
3. **每条注明来源**(链接)。
4. **给推荐,不穷举**;诚实说代价。

## 1. 覆盖范围:板上**所有**芯片,不只主控
按功能分层逐一列(有就列,没有就标无):
- 主控 SoC / 协处理器
- 内存 + 存储
- 摄像头 sensor + ISP
- IMU / 传感器(陀螺、地磁、气压、PPG…)
- 音频:codec + 功放(amp)+ MEMS 麦
- 唤醒词 / always-on 协处理器
- WiFi/BT(SoC 集成 or 外挂模组)+ GNSS
- 显示:光机(micro-OLED/LCoS/microLED)+ 驱动 IC(若有 HUD)
- 电源:PMIC + 升降压 + 电量计
- 充电盒 MCU(若有)

## 2. 每颗芯片:四轴评估
| 轴 | 看什么 |
|----|--------|
| 散热 | 发热量、封装、是否需导热设计 |
| 功耗 | 待机 mW + 活跃 mW |
| 重量 | 封装尺寸/克重 |
| 算力 | TOPS / DMIPS / 专用加速(无算力标 N/A) |

> **按 material 程度用四轴**:主控、光机、功放、唤醒词芯片要认真评四轴;IMU/PMIC/codec 等微小件,散热和重量直接标"可忽略",只记功耗(和有无算力)。不为凑格式编数据。

## 3. 商业维度
- **厂商**
- **可及性**:公开可买 / 需 NDA / 仅模组形式 / 仅 OEM
- **价格**:单价 + 百片估(查不到标"未公开")

## 4. 对齐判断(对每颗/每方案)
判定与我们目标**一致 / 出入**,一句话说清。
**我们的目标基线**:轻便(整机 <50g)、便宜、离线/自主(算力在眼镜本体)、可二次 PCB 塞进眼镜。

## 4.5 "别人为什么不用 / 我们为什么能用"(必答,防陷阱)
找到一颗看起来很香、但**量产眼镜没人用**的芯片时,**先问为什么没人用,再问我们凭什么能用**:
- **量产眼镜为什么不用它?** 通常有真实原因:不是眼镜芯片(无always-on枢纽/无AR栈/无参考平台)、持续功耗不符、缺生态/中间件拖慢上市、或它们的产品需求和我们不同。
- **我们凭什么能用?** 只有当**我们的产品需求确实不同**(如离线/触发式 vs 它们的连续+云端),且**我们愿意接受它们不接受的代价**(自建软件栈、外挂协处理器、较低打磨、原型更厚)时才成立。
- **反转检查**:如果"别人不用的原因"对我们同样成立(如我们其实也需要连续视觉/全程always-on),那我们也用不了——别自欺。
- 结论必须写成:**"它们不用因为X;我们能用因为我们的Y不同;但前提是Z(待验证)"**。

## 5. 输出格式
- **表格化**:芯片 × 功能 × 四轴 × 厂商/可及性/价格 × 对齐
- 末尾:**"我们该用 / 不该用 / 待定"** 的推荐 + 理由
- **来源链接清单**
- 查不全的地方,明确列"未查证项",不用猜的填满

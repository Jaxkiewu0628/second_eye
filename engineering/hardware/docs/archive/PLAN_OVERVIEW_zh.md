# 运动 AR 眼镜 — 原型方案概览

_交接文档。目标：在投入光学器件成本之前，先低成本验证视频管线。_
_日期：2026-06-22_

---

## 一句话总结（TL;DR）

我们要做一款运动 AR 眼镜（心率监测 + 类似高尔夫球轨迹的镜片叠加显示）。电子部分最难的，是在一块极小的板子上把实时视频**流畅、低延迟**地传出去。最初用的 ESP32-S3 做不到，因为它**没有硬件视频压缩**，而且共享的无线电带宽弱。解决方案是改用一颗小型 **Linux 摄像头 SoC（瑞芯微 RV1106 / "Luckfox" 开发板，约 $50）**，它能用硬件压缩视频，甚至能在设备端直接做目标追踪。**阶段 0 = 买板子，用 WiFi 推 1080p 视频流，测延迟。** 这个数字好，整个架构就验证通过了。

---

## 驱动一切的核心思想

> **不要传输原始像素。在设备端压缩或分析，然后只发送很少的数据。**

| 策略 | 发送什么 | 带宽 | 何时用 |
|---|---|---|---|
| **只发结果** | 球的位置、轨迹、心率 | **~10–100 kbps** | 最佳 — BLE 就够 |
| **设备端压缩** | H.265 视频流 | **2–5 Mbps** | 手机需要画面时 |
| ~~发原始 / MJPEG~~ | 未压缩帧 | 50–1000 Mbps | ❌ ESP32 被迫走的路 |

RV1106 三种都能做；ESP32-S3 只能做最差的那种。

## 外形与尺寸 — 能装进眼镜吗？

**简短答案：*芯片*能装下；*开发板*和*模组*装不下。** 进眼镜需要定制 PCB，而真正的尺寸/重量瓶颈是电池、光学和散热 —— 不是 SoC。

**尺寸阶梯：**

| 形态 | 尺寸 | 能塞进镜腿吗？ |
|---|---|---|
| Luckfox 开发板 | 数厘米 + 排针/接口 | ❌ 只能放桌面 |
| Core1106 SoM（模组） | **30 × 30 mm** | ❌ 太大（为紧凑摄像头/IoT 设计） |
| 裸 RV1106 芯片 | **~10–15 mm** SiP（芯片 + 内存同封装） | ✅ 仅限**定制柔性 PCB** |

_（即便是较粗的智能眼镜镜腿，内部宽度也只有约 6–10 mm。）_

**真实眼镜怎么做（Meta Ray-Ban Display 拆解）：**
- 元件**分布在两条镜腿**上 —— 一条放电池 + 扬声器 + 麦克风；另一条放主板（Snapdragon AR1 + 闪存）。
- **柔性 PCB 经铰链**走线到前框。
- 电池 = **960 mWh（约 260 mAh）**；整机重量 = **69 g**（普通 Ray-Ban Meta 约 49 g；舒适目标约 40–50 g）。

**真正让眼镜"笨重"的因素（不是 SoC）：**
1. **电池** —— 约 0.5–1 W 的实时视频，配 ~960 mWh 电池 ≈ **1–2 小时**续航。
2. **显示引擎 + 光学**占据前框大部分体积。
3. **散热** —— 密闭镜腿内约 0.5–1 W，贴着皮肤会发热。
4. **配重** —— 对抗约 50 g 的舒适重量预算。

**由此倒逼的架构决策 —— 算力放在哪里？**
- **一体式**（算力 + 电池 + 显示全在镜架内）：最重、最热、续航约 1–2 小时。
- **眼镜 + 有线计算盒（建议评估）：** 把 RV1106 + 大电池放进**口袋/夹扣计算盒**；眼镜只保留摄像头 + 微显示 + 传感器 + 一根细线 → 大幅更轻更凉。很适合户外运动。
- **眼镜 + 手机：** 镜架内体积最小；代价是增加无线延迟。

## 关键数字

**不同视频格式所需带宽**（这正是我们放弃 ESP32 的原因）：

| 格式 | 720p @30 | 1080p @30 |
|---|---|---|
| 原始（未压缩） | 442 Mbps | ~1 Gbps |
| MJPEG（ESP32 唯一选项） | 12–24 Mbps | 30–50 Mbps |
| **H.264**（RV1106 硬件） | 1–4 Mbps | 4–8 Mbps |
| **H.265**（RV1106 硬件） | 1–3 Mbps | 2–5 Mbps |

→ 用 H.265，**1080p 占用的带宽比 ESP32 推 VGA 还低。**

**延迟目标（端到端 / glass-to-glass）：** 流式传输 **< 150 ms**。高尔夫轨迹弧线对延迟较宽容；头部锁定叠加才需要 < ~20 ms（非阶段 0 的关注点）。

**无线现实：** 真正的瓶颈是 WiFi 实际吞吐，而非芯片。压缩后的 H.265（2–5 Mbps）余量很大；原始视频在任何可穿戴无线电上都塞不下。

## 数据通路

```
摄像头 ──MIPI-CSI──> [ ISP：图像处理 ] ──> [ 硬件 H.265 编码 ] ──> WiFi ──> 手机/电脑
                                       └──> [ NPU：检测球/追踪 ] ──> 发坐标（~kbps）
```

左侧（摄像头→ISP→编码→WiFi）**现成可用，一个下午就能跑通。**
NPU 分支（设备端追踪）是**阶段 1**。

## 构建阶段与验收标准

| 阶段 | 目标 | 完成标志 | 时间 |
|---|---|---|---|
| **0. 管线验证** | 用 WiFi 推硬件压缩视频 | 测得 **1080p30 H.265 @ 2–5 Mbps，延迟 <150 ms** | 1–2 周 |
| **1. 设备端追踪** | 在板上检测球/目标 | 先经典 CV，再 NPU（YOLO-nano）稳定检测；高尔夫 = 捕捉击球瞬间 + 物理公式算弧线 | 3–4 周 |
| **2. 心率** | 太阳穴/鼻梁 PPG + 运动滤波 | 运动中与 Polar H10 胸带对比在容差内 | 并行 |
| **3. AR 显示** | 把叠加画面送到眼前 | 叠加可见且清晰（先用现成 AR 眼镜当显示器） | 6–12 周 |
| **4. 形态落地** | 缩小到可穿戴板 | RV1106 SoM 上定制 PCB，电池、散热 OK | 第 4 个月起 |

## 物料清单（原型）

| 项目 | 约成本 |
|---|---|
| Luckfox Pico Ultra W + 摄像头 | $40–60 |
| 心率（PPG）传感器 + IMU | $15–40 |
| AR 显示 — *原型取巧*（用 Xreal/Rokid 眼镜当显示器） | $200–400 |
| AR 显示 — *真集成模组*（birdbath 或 波导+microLED） | $300–3000+ |
| 电池 / 供电 / 杂项 | $20–50 |

**电子原型（不含显示）：约 $80–150。** **成本由显示/光学主导**，而非计算单元。（量产 BOM 中 SoC 仅约 $8–15 —— 选 Linux 并不会让产品变贵。）

## 主要风险（按优先级）

1. **AR 显示/光学** —— 采购、成本、户外阳光下亮度。最大未知数。*缓解：单色 microLED HUD（文字 + 轨迹弧线）便宜且阳光下可读 —— 我们很可能不需要全彩 AR。*
2. **头戴式追球**确实难（目标小、快、且摄像头在动）。*缓解：捕捉击球瞬间 + 物理公式，不要全程视频追踪。*
3. **RV1106 CPU 占满风险** —— 同时做 CV + 编码 + 推流 + 显示（单 A7 核）。
4. **厂商 SDK 成熟度** —— 瑞芯微文档不如树莓派丰富。
5. **散热 / 电池** —— 在镜腿这种小空间内。

## 软件栈（参考）

- **系统：** Luckfox **Buildroot** 镜像（精简；自带 RTSP 摄像头示例）。需要 `apt` 时才上 Ubuntu 镜像。
- **视频编码：** 瑞芯微 MPP（硬件 H.264/H.265）
- **推流：** 自带 `rkipc` RTSP（`rtsp://<板子IP>/live/0`），或 GStreamer/WebRTC
- **设备端 AI：** RKNN-Toolkit2（转模型 → 在 NPU 上跑）
- **视觉：** OpenCV

## 第一步行动（本周）

1. 下单 **Luckfox Pico Ultra W + 配套 MIPI 摄像头**（约 $50）。
2. 烧录 **Buildroot** 镜像（SocToolKit → 选 RV1106 → SD 卡）。
3. 开机，SSH 登录（`pico` / `luckfox`），用播放器打开 `rtsp://<板子IP>/live/0`。
4. **测量并记录：分辨率/帧率、码率、端到端延迟。**

这个延迟 + 码率数字决定架构是否可行 —— 而验证它只需 $50。

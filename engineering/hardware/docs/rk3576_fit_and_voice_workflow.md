# RK3576 契合度 + 本地 STT/TTS 工作流程分析

> 客观评估,结合我们的应用场景 · 2026-06-28

---

## 一、先定义应用场景(再谈契合)

任何"芯片好不好"的判断都脱离不了场景。我们的场景是:

| 维度 | 我们的定位 |
|------|-----------|
| 产品 | AR 运动眼镜(高尔夫轨迹/目标追踪/语音辅助) |
| 网络环境 | **离线/弱网为常态**(球场、山野、户外) |
| 计算形态 | **全自主 standalone**(计算在眼镜本体,非纯手机中继) |
| 团队 | 学生团队,无融资,验证阶段 |
| 硬件约束 | 芯片须**可购买、可二次 PCB、有量产路径** |
| 功能基线 | 本地视觉(YOLO)+ 本地 STT/TTS + 云端 LLM,留 5× headroom |

> 这套场景里有两个关键词决定了一切:**离线**和**自主**。下面所有结论都围绕它们。

---

## 二、为什么 RK3576 契合

### 2.1 硬指标对照

| 需求 | RK3576 提供 | 是否满足 |
|------|------------|:---:|
| 算力(视觉+语音+5×余量,需 ~5-6 TOPS) | **6 TOPS NPU** | ✅ |
| 本地 STT/TTS 部署 | **glibc**(Ubuntu/Debian),sherpa-onnx 可 pip install | ✅ |
| 1080p@60fps 录像 | 硬件 H.265 编码(不占 NPU) | ✅ |
| 可购买 | 立创 ¥165,开发板 ¥300-600 | ✅ |
| 量产路径 | 同款 SoC 有 SoM 模块,可直接设计底板嵌入眼镜 | ✅ |
| WiFi/BT | 外挂模组可达 WiFi 6 + BT 5.3 | ✅ |

### 2.2 实测数据背书(非纸面参数)

均来自真实 RK3576 硬件的公开实测(rkvoice-stream、MYIR、鲁班猫3 实测):

| 任务 | RK3576 实测 | 我们的需求 | 余量 |
|------|:---:|:---:|:---:|
| STT(Paraformer hybrid) | RTF 0.29(快 3.4×) | 快于实时 | 充足 |
| TTS(Matcha+Vocos) | RTF 0.13,首音 320ms | 快于实时 | 充足 |
| YOLOv8n 视觉检测 | 30fps 真实 / 90fps 纯推理 | ≥10fps | 3-9× |

### 2.3 为什么不是别的芯片

| 候选 | 否决原因 |
|------|---------|
| **Snapdragon AR1** | 买不到(需 Qualcomm NDA),成本占整机 BOM 37-50% |
| **RV1106**(旧板) | 单核 A7 + uclibc,STT 慢 57×,语音链路完全不可用 |
| **RK3588** | 8 TOPS 但 3-5W,一体式眼镜散热/功耗扛不住 |
| **RK3576** | **算力够、可购买、glibc 生态、有量产路径——唯一可及的平衡点** |

### 2.4 契合的本质

RK3576 之所以契合,不是因为它最强,而是因为它**恰好落在我们场景的约束交集里**:
- 离线场景 → 需要本地算力 → 6 TOPS 够用
- 自主形态 → 需要嵌入眼镜 → 有 SoM、可二次 PCB、功耗可控
- 学生团队 → 需要可购买可量产 → ¥165 立创直购、中文生态成熟

> 选 RK3576 的那一刻,我们就已经为"本地能力"付了费(更贵更热的芯片)。这笔投入只有在真正使用本地 STT/TTS/视觉时才回本。

## 三、本地 STT/TTS 工作流程

### 3.1 整体数据流

```
                    ┌─────────────── 眼镜本体(RK3576)───────────────┐
   用户语音 ──麦克风──→ 本地 STT ──→ 意图判断 ──┬──→ 本地命令(离线)
                       (Paraformer)            │     ·记分 ·设备控制
                                               │     ·本地视觉问答 ·音乐
                                               │
                                               └──→ 需要 AI? ──文字──→ 云端 LLM
                                                                        │
                    本地 TTS ←──────────────────────文字←──────────────┘
                    (Matcha/Piper)
                       │
                    喇叭 ──→ 用户听到回应
```

**关键设计:STT/TTS 是统一的语音 I/O 层,LLM 只是它背后的消费者之一。**

### 3.2 命令分流(决定离线可用范围)

| 语音命令类型 | 是否需要云端 LLM | 离线可用 |
|-------------|:---:|:---:|
| "记录这一杆" / "第几洞" / "总杆数" | ❌ 本地状态 | ✅ |
| "拍照 / 录像 / 暂停音乐" | ❌ 设备控制 | ✅ |
| "前面旗杆多远" | ❌ 本地视觉(YOLO+测距) | ✅ |
| "给我打球建议" / 开放对话 | ✅ 云端 LLM | ❌ 需网络 |

本地命令理解甚至**不需要 LLM**,STT + 小型意图分类器(关键词级)即可。

### 3.3 实测延迟(完整链路)

```
本地命令(无 LLM):
  STT(~1.5s)+ 本地处理(<0.1s)+ TTS首音(~0.3s) ≈ 1.9s  全程离线

AI 对话(走云端 LLM):
  STT(~1.5s)+ LLM首token(网络决定)+ TTS首音(~0.3s)
  └ 只上传文字(几十字节),不上传音频 → 弱网下更容易成功
```

### 3.4 模型选型(有坑)

| 环节 | 推荐模型 | 注意 |
|------|---------|------|
| STT | Paraformer-zh-small / SenseVoice | 非自回归,比 Whisper 快 3-5×,中文更好 |
| TTS | **vits-icefall-zh-aishell3 / Matcha / Piper** | ⚠️ 多数 vits-zh 在 ARM 上 RTF 1.6-4.3(慢于实时),**选错致命** |

## 四、优缺点(客观)

### 4.1 优点(结合场景)

| 优点 | 对我们场景的意义 |
|------|----------------|
| **离线可用** | 球场/山野没网时,本地命令(记分、控制、视觉)照常工作 |
| **延迟低** | STT/TTS 不走网络,本地命令 ~1.9s 完成;AI 对话只传文字不传音频 |
| **隐私** | 音频不出设备,只有几十字文本上云 |
| **规模零边际成本** | 用户增长不增加语音 API 费(对比云端线性增长) |
| **省流量/省电** | 弱网下不上传音频流,WiFi 不需长时间开启 |
| **算力已付费** | 既然买了 6 TOPS 的芯片,本地跑才不浪费 |

### 4.2 缺点 / 代价(诚实,不回避)

| 缺点 | 真实影响 |
|------|---------|
| **研发投入** | 本地部署/量化/调优要数月;云端 API 一行代码 |
| **模型出厂冻结** | 升级模型需 OTA 推送固件;云端可服务端随时换 |
| **质量天花板** | 极端口音/噪声/多语种,云端大模型仍更稳(中文场景已基本追平) |
| **硬件更重更热** | 为了能本地跑,选了更贵更热的 SoC,眼镜重量/续航为此买单 |
| **不能让 LLM 功能离线** | 本地 STT/TTS 只让**非 LLM 命令**离线,AI 对话没网照样断 |

### 4.3 一个必须说清的边界

**本地 STT/TTS 的价值,正比于"不需要 LLM 的交互"占比:**
- 若产品多为命令/记分/控制/本地视觉 → 本地 STT/TTS 价值巨大
- 若产品几乎全是"问 AI 开放问题" → 云端 STT/TTS 跟 LLM 一起走更省事,本地意义有限

> 这是产品交互设计问题,不是纯技术问题。需先明确用户语音中命令 vs 开放对话的比例。

---

## 五、结论

1. **RK3576 契合我们的场景**——离线 + 自主 + 可量产的约束交集里,它是唯一可及的平衡点,且已有真实硬件实测背书。

2. **推荐混合架构**:本地 STT/TTS(离线 I/O 层)+ 云端 LLM(开放对话)。音频不出设备,只有文字上云,既保离线基础功能又借云端大脑。

3. **代价要认清**:研发投入、模型需 OTA、硬件更重更热是真实的账。本地不是"免费",是把成本从运营前移到了硬件和研发。

4. **下一步验证**:YOLOv8n→RKNN 转换链路打通(进行中),板子到货后做真机 `eval_perf` 与功耗实测,确认 1080p 场景下的真实帧率与续航。

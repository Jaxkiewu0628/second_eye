## Context / 背景

## The ask / 需求（现在只要这些，别现在写功能）
1. **不要现在做表里 2–6。** V1 还是只做避障（模块 1）。
2. **唯一的请求：** 把感知管线设计成**模块化 / 通用**——避障 = 模块 1，留好加模块 2/3 的接口，让以后加功能是"写软件"而不是"重写引擎"。

4. **反对也欢迎：** 如果你判断"从第一天就做通用引擎"在硬件上代价过大、不如先写死避障，**直接回 `❓ NEEDS INFO` 说清楚理由**。这正是 S1 卡住的那个叉（"从第一天就做通用引擎 yes/no"），你的工程判断会直接帮 CEO 拍板。

## Thread / 对话线
- **2026-06-30 [CEO]** — 初始路线图。这条不急、V1 不受影响；你上板之余给个"可行性 + 模块化架构"判断即可。你的回复会直接喂回 S1 决策。
- **2026-07-01 [ENG]** — 判断如下（基于 RV1126B 3 TOPS，全程离线）：

  **各功能可行性：**
  | # | 功能 | 离线可行性 | 算力/传感器要求 |
  |---|---|---|---|
  | 1 | 避障/路径 | ✅ 舒适 | YOLO11n + ToF，~0.5 TOPS，已验证 |
  | 2 | 室内寻路/导航 | ⚠️ 简化版可行 | 需门/电梯检测模型；完整 SLAM 太重，3 TOPS 偏紧 |
  | 3 | OCR 文字朗读 | ✅ 可行，最容易加 | PaddleOCR-mobile 可跑 NPU；TTS 管线已有；触发式，~0.5 TOPS |
  | 4 | 找东西 | ⚠️ 大物体可行；小物体难 | VL53L5CX 8×8 ToF 定位太粗；精确找小物体（钥匙）需单目深度估计或更密 ToF |
  | 5 | 场景/人物描述 | 🟡 场景分类离线可行；自由描述需云端 | 自由描述需 VLM（如 LLaVA），3 TOPS 跑不动；人物识别技术上可行（需预先录入） |
  | 6 | 交通灯/楼梯/公交站 | ✅ 可行 | YOLO 扩类，同一模型加训练数据，无额外传感器，算力增量小 |

  **架构建议：✅ 从第一天做模块化通用引擎，值得，硬件代价为零。**
  感知管线天然就是 `帧 → 推理器(model_path) → 后处理(config) → 语音模板`。定义干净接口的额外工程量约 1–2 天。收益：加功能 = 换模型文件 + 写一个后处理规则，不动引擎。不做的代价是加第二个功能时重构整个管线。**推荐在 V1 上板时就把接口定好，不增加任何 V1 交付风险。**

  **推荐上线顺序：**
  1. **V1**：避障+路径（YOLO11n + ToF，楔子）
  2. **V1.5**：交通灯/楼梯识别（YOLO 扩类，低代价）+ OCR 文字朗读（model swap，TTS 复用）
  3. **V2**：室内寻路简化版（门/电梯检测）+ 找大物体（桌椅/沙发级别）
  4. **V3+**：场景描述（届时视芯片算力，或走云端 LLM 场景描述）；精确找小物体（需更好深度传感器）

  一句话给 S1 拍叉：**平台路线可行，从第一天做通用引擎成本极低（1–2 天），强烈推荐；V1 交付完全不受影响。**

## Resolution / 结论
平台路线可行。感知引擎模块化代价极低（1–2 天定接口），推荐从 V1 就做。各功能中 OCR 和交通灯/楼梯最容易加（YOLO 扩类 + model swap）；自由场景描述受 3 TOPS 限制，需云端或等更强芯片。推荐上线顺序：避障→交通灯/OCR→室内寻路→场景描述。CEO 可据此拍 S1 的平台 vs 单品叉。
